Del Transformer al Chatbot

Un Transformer solo predice la siguiente palabra

  • Un LLM “base” (recién pre-entrenado) es, en el fondo, un autocompletado gigante: dado un texto, predice el siguiente token más probable, una y otra vez.

  • Eso lo hace muy bueno continuando texto, pero no necesariamente respondiendo.

  • Si le escribes “¿Cuál es la capital de Francia?”, un modelo base podría seguir con más preguntas (porque en internet las preguntas suelen venir en listas) en vez de contestar “París”.

Predecir la siguiente palabra ≠ ser un asistente útil. Falta alinear al modelo para que se comporte como un ayudante que responde lo que le pides, en el tono adecuado.

Tres etapas hasta el asistente

  1. Pre-entrenamiento: predecir el siguiente token sobre enormes corpus de texto. El modelo adquiere gramática, hechos y “conocimiento del mundo”, pero todavía no sabe comportarse como asistente.

  2. Fine-tuning supervisado (SFT / instruction tuning): se le muestran muchos pares (instrucción → buena respuesta) escritos por personas. Aprende el formato de “me piden algo y respondo de forma útil”.

  3. RLHF (Reinforcement Learning from Human Feedback): personas rankean varias respuestas del modelo; con esos rankings se entrena un modelo de recompensa, y luego se ajusta el LLM para producir lo que la gente prefiere (más útil, mejor tono, más seguro).

Ese ajuste (SFT + RLHF) es lo que convierte un autocompletador en un chatbot como ChatGPT, Claude o Gemini.

Sin álgebra

Aquí solo nos quedamos con la intuición. La idea clave: el mismo Transformer que predice tokens, afinado con ejemplos y preferencias humanas, se transforma en un asistente.

Mencionar que existen variantes más modernas de la última etapa (DPO, RLAIF) pero la intuición es la misma: optimizar hacia respuestas que las personas prefieren.

Cómo elegir LLM

No existe “el mejor modelo”

  • Existe el mejor modelo para tu tarea, considerando cuatro ejes:

    • Inteligencia / calidad en esa tarea concreta.
    • Costo (se paga por token).
    • Velocidad / latencia.
    • Privacidad (¿puedo mandar estos datos afuera?).

  • Un modelo enorme y carísimo es un desperdicio para corregir ortografía; un modelo chico se queda corto para razonamiento complejo.

La pregunta correcta no es “¿cuál es el mejor?” sino “¿cuál es el más barato y rápido que resuelve bien esta tarea?”.

LMArena: preferencia humana

  • LMArena (ex Chatbot Arena / LMSYS): le haces la misma pregunta a dos modelos anónimos y votas cuál respondió mejor.

  • Con miles de votos a ciegas se arma un ranking estilo Elo (como el ajedrez).

  • Mide algo difícil de capturar con benchmarks: qué tan útiles le parecen las respuestas a las personas.

  • Hay rankings por categoría (código, matemáticas, escritura, etc.).

🔗 lmarena.ai

Artificial Analysis: costo vs. inteligencia

  • Artificial Analysis agrega muchos benchmarks en un único Intelligence Index (un puntaje de “inteligencia”).

  • Lo más útil: grafica ese puntaje contra el precio y la velocidad (tokens/seg), así que de un vistazo ves el balance costo/calidad.

  • Ideal para responder: “de los modelos que son suficientemente buenos, ¿cuál es el más barato y rápido?”.

🔗 artificialanalysis.ai

Heurística práctica: usa el modelo más barato que sea “suficientemente bueno”. Reserva los modelos frontera (caros) para razonamiento difícil; usa modelos chicos y rápidos para tareas masivas y simples (como la corrección de texto que veremos al final).

Propietarios vs. open-weight

También importa si el modelo es propietario (solo vía API: GPT, Claude, Gemini) o de pesos abiertos (Llama, Qwen, DeepSeek, Gemma), que puedes descargar y correr tú mismo. Esto último es clave para la privacidad (lo vemos en un momento).

Buenas prácticas

Trabajar bien con LLMs

  • Contexto lo es todo: el modelo es stateless, en cada llamada le mandas todo el contexto que necesita. No “recuerda” la conversación anterior salvo que se la vuelvas a enviar.

  • Da instrucciones claras: define un rol (system prompt), explicita el formato de salida y, si ayuda, incluye ejemplos (few-shot). Garbage in, garbage out.

  • Temperatura baja (≈ 0) para tareas estructuradas, extracción o corrección (quieres consistencia, no creatividad).

  • Costo: pagas por token de entrada + salida. Inofensivo en una prueba, relevante al procesar miles de filas.

  • Alucinaciones: el modelo puede inventar con total seguridad. Verifica las salidas y mantén un humano en el loop.

  • Reproducibilidad: registra el prompt y la versión del modelo que usaste.

Protección de datos

Somos una institución pública: trabajamos con datos de personas y empresas.

Antes de enviar algo a una API externa

Pregúntate siempre: ¿qué dato es esto y dónde se va a procesar? Muchas APIs corren en servidores en el extranjero. No podemos mandar datos personales o confidenciales a la nube fuera del país.

  • Anonimizar / seudonimizar antes de enviar: quitar RUT, nombres, direcciones, folios. Si el modelo no necesita el dato identificador, no se lo mandes.

  • Modelos locales / on-prem para datos sensibles: corre el modelo en infraestructura propia, los datos nunca salen.

    • Ollama: fácil, para correr modelos open-weight en tu equipo.
    • vLLM: servidor para producción.

  • Lee las políticas de retención y de entrenamiento del proveedor (¿guardan tus prompts?, ¿entrenan con ellos?).

Marco legal en Chile: Ley 19.628 sobre protección de la vida privada y la nueva ley de datos personales. Ante la duda, trata el dato como sensible y consulta con gobierno de datos.

Recalcar: la regla de oro es que un dato identificable de una persona no debería salir hacia un servidor en el extranjero. Si se necesita procesar texto con datos sensibles, la vía es un modelo local.

OpenRouter

¿Qué es OpenRouter?

  • Una API única para hablar con cientos de modelos de muchos proveedores (OpenAI, Anthropic, Google, Meta, DeepSeek, …).

  • Una sola API key y una sola cuenta de facturación para todos.

  • Cambiar de modelo = cambiar un string ("openai/gpt-4o" → "anthropic/claude-3.5-haiku"). No reescribes el código.

  • Es compatible con la API de OpenAI, así que funciona con herramientas que ya conoces.

Perfecto para aprender y comparar: pruebas el mismo prompt en varios modelos sin crear una cuenta en cada proveedor. 🔗 openrouter.ai

Mostrar en vivo la página de openrouter.ai: el catálogo de modelos, los precios por millón de tokens y la página de cada modelo.

Conectarse — opción A: requests

La forma más “cruda”: un POST HTTP al endpoint. Útil para ver qué viaja realmente.

import os
import requests

# La key NUNCA se escribe en el código: se lee de una variable de entorno.
API_KEY = os.environ["OPENROUTER_API_KEY"]

respuesta = requests.post(
    url="https://openrouter.ai/api/v1/chat/completions",
    headers={"Authorization": f"Bearer {API_KEY}"},
    json={
        "model": "openai/gpt-4o-mini",            # <- el id del modelo es solo un string
        "messages": [
            {"role": "system", "content": "Eres un asistente conciso."},
            {"role": "user", "content": "Explica qué es el INE en una frase."},
        ],
    },
)

data = respuesta.json()
print(data["choices"][0]["message"]["content"])

Respuesta (ilustrativa):

# El Instituto Nacional de Estadísticas (INE) es el organismo público de Chile
# encargado de producir las estadísticas oficiales del país.

Conectarse — opción B: SDK de openai (recomendado)

Como OpenRouter es compatible con OpenAI, podemos usar su SDK y solo cambiar la base_url. Más limpio y maneja los detalles por nosotros.

# pip install openai
import os
from openai import OpenAI

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",   # <- apuntamos a OpenRouter
    api_key=os.environ["OPENROUTER_API_KEY"],
)

resp = client.chat.completions.create(
    model="anthropic/claude-3.5-haiku",        # <- cambiar de modelo = cambiar este string
    messages=[
        {"role": "system", "content": "Eres un asistente conciso."},
        {"role": "user", "content": "Explica qué es el INE en una frase."},
    ],
    temperature=0,
)

print(resp.choices[0].message.content)

El id del modelo (proveedor/modelo) lo eliges en openrouter.ai/models. Para tareas masivas conviene un modelo barato y rápido; lo demostramos a continuación.

La función de corrección

Una función que recibe un texto, arma el prompt y devuelve la versión corregida. Usamos temperature=0 para que sea consistente.

from openai import OpenAI
import os

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",
    api_key=os.environ["OPENROUTER_API_KEY"],
)

SYSTEM = (
    "Eres un corrector de textos en español. Corrige la ortografía y la "
    "gramática del texto del usuario. Devuelve SOLO el texto corregido, "
    "sin comillas ni explicaciones."
)

def corregir(texto: str) -> str:
    resp = client.chat.completions.create(
        model="openai/gpt-4o-mini",          # modelo barato: la tarea es simple
        messages=[
            {"role": "system", "content": SYSTEM},
            {"role": "user", "content": texto},
        ],
        temperature=0,
    )
    return resp.choices[0].message.content.strip()

# Aplicamos la función a cada fila -> nueva columna
df["texto_corregido"] = df["texto"].apply(corregir)
df

Escalar con cuidado

Pasar de 5 filas a 50.000 cambia las prioridades:

  • Costo: multiplica el costo por fila por el total. Estima antes de lanzar todo.
  • Rate limits: el proveedor limita llamadas por minuto → procesa por lotes y agrega pausas.
  • Reintentos: una llamada puede fallar (red, límite). Implementa reintentos con backoff.
  • Manejo de errores: decide qué hacer con una fila que falla (reintentar, marcar, omitir) sin botar todo el proceso.
  • temperature=0: consistencia y resultados reproducibles.
  • Salida estructurada (JSON): cuando necesitas más que texto plano (p. ej. {"corregido": ..., "cambios": [...]}), pide salida en JSON para parsearla directo.

Y, antes de todo: ¿estos textos contienen datos personales? Si es así, vuelve a la slide de protección de datos — anonimiza o usa un modelo local.
